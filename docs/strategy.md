# mcp-jp の戦略メモ

このリポジトリを「なぜ作り、どこで価値を出し、どう続けるか」の整理。
OSS として公開し、その先をビジネスに繋げるための前提を言語化しておく。

---

## 1. 立ち位置：何を作っているか

**日本の SMB 向け SaaS のうち、公式 MCP が存在しないものへのコネクタ集。**

- AI エージェント（Claude 等）から日本のクラウドサービスを自然言語で操作可能にする。
- 各コネクタは独立した pip インストール可能パッケージ。1 コネクタ ≒ 5 ツール前後。
- 重複を避けるため、公式 MCP が出たサービスは [`archive/`](../archive/) へ退避し、
  README で「公式を使え」と明示する。

この「公式不在のものだけをやる」制約が、リポジトリの一貫性と信頼の核。

---

## 2. 価値の所在：コネクタ本数ではない

率直に言うと、**コネクタの数そのものはほとんど堀にならない。**
REST API があれば実装は誰でも数時間で再現でき、公式 MCP が出れば即座に陳腐化する
（実際 30 以上を archive に退避した）。本数を競うのは消耗戦。

価値が乗るのは、コネクタの「上」にある運用基盤のレイヤー：

| レイヤー | 内容 | なぜ価値になるか |
|---|---|---|
| **認証・鍵管理** | OAuth フロー、トークン更新、複数テナントの鍵を安全に保持 | SMB が自前で用意できない。事故ると一発で信用を失う領域 |
| **ホスティング** | コネクタを常時稼働させ、Claude 等から即使える状態にする | ローカル `pip install` は導入障壁が高い。Remote MCP 化で解消 |
| **監査・権限** | 誰がどのツールで何を操作したかのログ、書き込み系の承認フロー | 業務利用＝統制要件。ここが SMB の導入判断を分ける |
| **キュレーション** | 「公式不在で需要のあるものだけ」を選び続ける目利き | [候補選定マトリクス](./connector-candidates.md)と[巡回チェック](../tools/README.md)で仕組み化 |

OSS のコネクタ集は**この基盤の入口（信頼とリード獲得）**として機能させる。
無料で実装を配り、運用の面倒（鍵・ホスティング・統制）を引き受けるところで対価を得る。

---

## 3. OSS 公開の狙い

公開はマネタイズの否定ではなく、その前段：

1. **信頼の証明** — 動くコード・テスト・CI を公開し、「日本 SMB の MCP に一番詳しい」
   ポジションを可視化する。
2. **リード獲得** — 「公式不在のあのサービスも繋ぎたい」という需要が Issue/問い合わせで集まる。
   これは有償の運用基盤・受託のリードそのもの。
3. **キュレーションの外部化** — 候補や「公式が出た」情報が他者からも入る。巡回コストが下がる。

公開する以上、**実装の質（テスト・エラー処理・README）が看板**になる。
だから本数より「載せたものは全部ちゃんと動く」を優先する。

---

## 4. 収益化レイヤーの仮説

近い順に：

1. **受託コネクタ開発** — 「自社で使っている公式不在 SaaS を繋ぎたい」企業への個別実装。
   OSS が実績兼ポートフォリオになる。最も早く回収できる。
2. **ホスティング / Remote MCP** — 鍵管理・常時稼働・監査ログ込みで、コネクタを SaaS として提供。
   `pip install` の障壁を取り除く対価。
3. **キュレーション付き提供** — 「日本 SMB 向けに検証済みコネクタ束」をまとめて提供・サポート。

いずれも前提は「公式不在の見極め」と「動くことの保証」。そこは OSS 側の仕組み
（[マトリクス](./connector-candidates.md) + [巡回チェック](../tools/README.md) + テスト/CI）で担保する。

---

## 5. 続けるための歯止め

戦略が崩れる最大要因は **「公式 MCP が出たのに独自実装を抱え続ける」** こと。
これを人手の記憶に頼らない：

- **[候補選定マトリクス](./connector-candidates.md)** — 着手前に「公式不在の持続性／需要／検証可能性／
  日本 SMB 密着度」でスコアリング。公式が出そうなものは最初から作らない。
- **[公式MCP巡回チェック](../tools/README.md)** — 稼働コネクタが公式提供済みと重複したら CI で落ちる。
  四半期ごとに一覧を見直し、`archive/` へ退避する。

「本数を増やす」ではなく「載っているものの信頼を保つ」方向に運用を寄せる。

---

## 6. リスクと割り切り

- **公式 MCP がほとんどの主要 SaaS に行き渡ると、独自実装の対象は痩せる。** → 価値を
  コネクタ本体ではなく運用基盤・受託に置いているので、対象が減っても基盤の需要は残る。
- **書き込み系ツールの事故リスク。** → 業務データを壊す操作は監査・承認のレイヤー（有償側）の
  責務とし、OSS 側は読み取り中心＋明示的な確認で設計する。
- **個人運用の持続性。** → 本数を追わないことで維持コストを抑える。テスト/CI で
  「壊れていないこと」だけは自動で守る。

---

## 7. 露出チャネル（掲載ディレクトリ）

OSS をリード獲得の入口にするため、MCP の掲載ディレクトリに登録する。

- **mcp.so** — リポジトリ URL を貼るだけで一覧化できる。登録済み（2026-06）。
- **PulseMCP** — 直接フォーム登録は廃止され、現在は「Official MCP Registry に publish → 毎日取り込み」
  か「`hello@pulsemcp.com` にメール依頼」の2択。Official Registry は 1 サーバー＝1 `server.json` 前提で
  モノレポと噛み合わないため（smithery と同じ壁）、メールで「リポジトリを1エントリとして掲載」を依頼（2026-06 送信済み・返信待ち）。
- **glama.ai** — 当初「`awesome-mcp-servers` への掲載（PR #8647）から自動同期される」と想定していたが、
  実際は**逆順**だった（2026-07 判明）。awesome へ PR を出すと glama-check bot が
  「(1) glama.ai に submit し Dockerfile でチェック通過 → (2) スコアバッジを PR に追記」を要求し、
  これを満たさないとマージされない。チェック条件は「Dockerfile でサーバーが**起動し introspection
  (`tools/list`) に応答**」のみで、実 API 疎通は不要。各コネクタの `list_tools()` は認証情報未設定でも
  ツール一覧を静的に返す設計のため、環境変数無しで introspection が通る。
  Glama はモノレポの各サブディレクトリを別サーバーとして個別登録できるため、smithery のような
  モノレポの壁には当たらない（ただし公式提出フォームに GitHub リポジトリ URL しか入力欄がなく、
  サブディレクトリ指定はできないため、Description 欄に対象サブディレクトリと Dockerfile の場所を
  明記して手動レビューでの認識に委ねる運用）。
  → 全 33 コネクタに Dockerfile 作成済み、`glama.json`（maintainers）追加済み。
  まず `akashi` 1件のみ試験的に submit 済み（2026-07・審査待ち）。
  残タスク: akashi の審査結果（モノレポのサブディレクトリが正しく個別サーバーとして認識されるか）を
  確認してから、残り 31 コネクタを同様に submit → 全件通過後にバッジを PR #8647 に追記。
- **smithery.ai は見送り（2026-06 判断）。** smithery は「1 リポジトリ = すぐ動かせる 1 サーバー」を
  前提とするが、mcp-jp は独立 pip パッケージのモノレポで噛み合わない。露出は上記 3 つで十分なため、
  構造を smithery 向けに作り替えるコストは見合わないと判断。将来 smithery がモノレポ対応したら再検討。
