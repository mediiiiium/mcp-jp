# コネクタ候補 選定マトリクス（叩き台）

新しいコネクタを「API があるから」で量産すると、(1) 公式 MCP に追い抜かれ、(2) 需要のない閲覧系を作り、(3) 鍵が取れず検証もできない、という消耗に陥る。
**着手前にこのマトリクスでスコアリングし、上位だけ作る。**

## スコアリング軸（各 0〜3 点、合計 12 点満点）

| 軸 | 0 点 | 3 点 |
|---|---|---|
| **A. 公式MCP不在の持続性** | 公式 MCP が既にある／半年内に出る兆候 | 日本SMB特化・ベンダーがAIに消極的で当面来ない |
| **B. 業務需要の強さ** | 単なる閲覧・ニッチ | 勤怠/請求/採用/在庫/CSなど日次業務でAI操作の価値が明確 |
| **C. API & 検証可能性** | API非公開／鍵が取れず疎通検証不能 | 安定したREST API公開＋開発者がテスト用鍵を取得可能 |
| **D. 日本SMB密着度** | グローバル大手（公式が必ず来る） | 国内SMBに広く普及し英語圏が作らない |

**判定:** 合計 **9 点以上を着手候補**、6〜8 は保留、5 以下は見送り。
A が 0（公式あり/直近GA）なら**他が高くても見送り**（追い抜かれ確定のため）。

## 運用ルール

1. **着手前に必ず Web で裏取り**: ①公式 MCP の有無（ベンダー発表・GitHub・`mcp.<service>.com`）②API ドキュメントの実在③テスト用クレデンシャル取得可否。
2. **公式MCP一覧の鮮度管理を自動化する**（README の表は手動だと必ず腐る）。月次で公式 MCP の新規提供をチェックし、出たコネクタは `archive/` へ移す。→ `scripts/` に巡回スクリプトを置く想定。
3. **需要シグナルはコードでなく市場から引く**: 対象SaaSのコミュニティ/フォーラム/X で「ChatGPT/Claude で〇〇したい」の声、各社の API 利用事例。
4. 着手したら本表に行を追加し、スコアと裏取り日・出典を残す。
